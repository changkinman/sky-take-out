# 苍穹外卖 - 日志规范与接口响应文档

## 一、 统一接口响应标准 (Response Signal)

为了保证前后端联调的一致性，所有 API 必须遵循统一的响应结构：

### 1. 响应结构体 (Result)
```json
{
  "code": 1,
  "data": { ... },
  "msg": "操作成功"
}
```

### 2. 状态码双轨制规范

我们将信号分为 **HTTP 传输信号**（网络层）和 **Business 业务信号**（数据层）：

#### A. HTTP 状态码 (Network Layer)
| 状态码 | 含义 | 触发场景 |
| :--- | :--- | :--- |
| **200** | **OK** | 请求成功送达后端并处理（无论业务成功或失败） |
| **401** | **Unauthorized** | JWT 令牌缺失、过期或签名错误 |
| **403** | **Forbidden** | 权限不足（如非管理员尝试访问管理接口） |
| **404** | **Not Found** | 请求路径错误，或者请求了不存在的资源 |
| **500** | **Internal Error** | 后端代码抛出未捕获异常（如空指针、SQL语法错误） |

#### B. 业务状态码 (Business Layer - Body.code)
| 状态码 | 含义 | 备注 |
| :--- | :--- | :--- |
| **1** | **SUCCESS** | 业务逻辑执行成功 |
| **0** | **ERROR** | 业务逻辑失败（如：余额不足、密码错误） |

### 3. 常见异常信号说明 (Exception Signals)
- **业务类 (code:0)**:
  - `账号已存在`: 当新增员工/分类重名时抛出。
  - `账号已禁用`: 登录时检测到 `status=0`。
  - `分类下关联了菜品`: 删除分类前的预检逻辑。
- **系统类 (HTTP 500)**:
  - `GlobalExceptionHandler`: 项目配置了全局异常处理器，会将 500 错误包装成 `code:0` 并返回具体的错误 `msg`。

## 二、 日志分级与输出规范

项目采用 **SLF4J + Logback** 进行日志管理，日志文件按天滚动存储。

### 1. 日志级别定义
- **INFO**: 记录核心业务流程。
  - 用户登录成功、订单支付回调、导出报表成功。
- **DEBUG**: 记录 SQL 执行详情及 AOP 增强逻辑。
  - 仅在 `dev` 开发环境下开启。
- **WARN**: 记录非致命性异常。
  - 密码输入错误次数过多、库存预警。
- **ERROR**: 记录系统级异常。
  - 数据库连接超时、第三方支付接口调用失败、空指针异常。

### 2. 输出位置
- **控制台**: 实时查看 SQL 及异常堆栈。
- **本地文件**: `logs/sky.log` (包含 info 级别以上日志)。
- **错误文件**: `logs/sky_error.log` (仅包含 error 级别日志)。

---

## 三、 JWT 鉴权日志
系统在 `JwtTokenAdminInterceptor` 中拦截所有 `/admin/**` 请求：
- **拦截成功**: 日志记录 `Current EmpId: {id}`。
- **拦截失败**: 日志记录 `JWT Check Fail: {reason}`，并返回 `401` 信号。

---

## 四、 AOP 自动填充日志
- 每次触发 `@AutoFill` 时，日志会记录被修改的 Entity 类名及操作类型（INSERT/UPDATE），确保数据一致性可追踪。
